The Test Planner is a **Claude Code plugin** that takes any web application codebase and produces a complete, production-ready E2E test suite in four steps. Install it once, run a single command, and the plugin handles orchestration, validation gates, and user confirmation checkpoints automatically. The final output is a set of test files ready to upload to Autonoma, backed by a live Environment Factory endpoint that creates isolated test data for every run.

## Installation

<InstallTabs>

Install the plugin from the marketplace:

```bash
/plugin marketplace add Autonoma-AI/test-planner-plugin
```

Then register it with Claude Code:

```bash
/plugin install autonoma-test-planner@autonoma
```

Then run the plugin:

```bash
/autonoma-test-planner:generate-tests
```

</InstallTabs>

The plugin runs each step in an **isolated subagent** and validates every output with **deterministic shell-script validation** (Python + YAML parsing) - not LLM-based checks. If validation fails, the agent sees the exact error and must fix it before proceeding. PostToolUse hooks validate every file write automatically, and cross-file consistency checks ensure outputs like INDEX.md test counts match features.json feature counts.

## Before you start

The plugin runs primarily against your **frontend codebase** - that's where the agent discovers pages, flows, and UI patterns. However, the current Step 2 flow also requires a working **Environment Factory endpoint** in your backend so it can fetch the SDK `discover` artifact before planning scenarios.

Before you start, make sure you have:

- access to the frontend codebase
- access to the backend codebase, or at least a running Environment Factory endpoint
- these environment variables in the Claude Code session:
  - `AUTONOMA_SDK_ENDPOINT`
  - `AUTONOMA_SHARED_SECRET`

These values should come from your project's Autonoma SDK integration. `AUTONOMA_SDK_ENDPOINT` is the full SDK endpoint URL in your app, and `AUTONOMA_SHARED_SECRET` is the shared HMAC secret configured for that integration.

If your frontend and backend are in the same monorepo, you're all set. If they're in separate repositories, make sure the frontend workspace can still reach the backend endpoint used for `discover`, `up`, and `down`.

## The four steps

### Step 1 - Generate a knowledge base

The agent analyzes your frontend codebase and produces `AUTONOMA.md`: a user-perspective guide to every page, flow, and interaction in your application. It also generates `skills/` files - step-by-step navigation guides the test runner uses to reach specific pages and perform setup actions.

**Consumes:** Your codebase.
**Produces:** `autonoma/AUTONOMA.md` + `autonoma/skills/*.md`

### Step 2 - Generate test data scenarios

The agent fetches the SDK `discover` response from your Environment Factory endpoint, saves it as `autonoma/discover.json`, and then designs three named test data environments: `standard` (realistic variety for most tests), `empty` (for empty state testing), and `large` (for pagination and performance). `scenarios.md` still records concrete values and relationships, but it now also records schema metadata and generated-value placeholders for fields that must stay dynamic.

**Consumes:** Knowledge base from Step 1 + `discover` output from your Environment Factory + relevant code context.
**Produces:** `autonoma/discover.json` + `autonoma/scenarios.md`

### Step 3 - Generate E2E tests

The agent produces an exhaustive set of test cases as natural language markdown files. Tests are distributed across tiers: core flows get 50-60% of coverage, supporting flows get the rest. The suite includes happy paths, input validation, state persistence, navigation, and cross-flow journey tests. An adversarial review agent runs after to find gaps.

**Consumes:** Knowledge base + scenarios from Steps 1-2.
**Produces:** `qa-tests/` directory with test files + `INDEX.md`

### Step 4 - Implement and validate the Environment Factory

The agent works against your backend to implement or complete the Environment Factory, then validates the planned scenarios against that implementation. It makes sure `discover`, `up`, and `down` work with the current SDK contract, smoke-tests the lifecycle, and persists approved scenario recipes for later automation.

**Consumes:** `discover.json` + scenarios from Step 2 + your backend codebase and/or running Environment Factory.
**Produces:** backend Environment Factory changes + `autonoma/scenario-recipes.json`

## How the steps connect

- **Step 1** output feeds Steps 2 and 3 - the knowledge base tells both agents what entities exist and what the core flows are
- **Step 2** output feeds Steps 3 and 4 - tests reference scenario data by name, and Step 4 uses the scenario plan to implement and validate the backend lifecycle
- **Step 3** output is what you upload to Autonoma - the test suite
- **Step 4** output is what makes those tests runnable - the Environment Factory plus the validated recipe layer that confirms `up` and `down` can create and clean up the planned data

## How validation works

Unlike prompt-based validation, the Test Planner uses **deterministic shell-script validators** at every step:

- **PostToolUse hooks** run after every file write, catching structural issues (missing frontmatter, invalid YAML, wrong file locations) immediately
- **Step-level validators** run Python and YAML parsing scripts to verify the complete output before the next step begins
- **Cross-file consistency checks** ensure inter-file references are correct - for example, INDEX.md test counts must match the actual number of test files, and features.json feature counts must align with the knowledge base

If any validation fails, the agent receives the exact error message and must fix the issue before the plugin allows it to proceed. This means you never end up with a broken intermediate output feeding into the next step.

## Review checkpoints

The plugin pauses between Steps 1-3 and asks for your review before the output is consumed by the next step. These are not optional - getting them right determines the quality of the final test suite.

| After step | What to review | Why it matters |
|------------|----------------|----------------|
| Step 1 | Core flows identified | Determines 50-60% of test coverage weight. Wrong core flows = poorly prioritized tests. |
| Step 2 | Scenario entity data + variable fields | Fixed values become direct assertions, and generated values become placeholders. Wrong names, counts, relationships, or variable markings = brittle tests. |
| Step 3 | Journey + Critical test samples | Confirms assertions reference real UI text, not vague descriptions. |
| Step 4 | Environment Factory implementation plan | Ensures the backend integration, secrets, lifecycle wiring, and scenario validation path are correct before code is written. |

Each step page explains what to look for and why it matters in detail.
